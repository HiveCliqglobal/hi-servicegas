Discovery · what we found

From fragile n8n to production stack.

A full audit of the live n8n bot — every state, every code node, every external call — surfaced real bugs, hardcoded secrets, and architectural fragility. The brief is bigger than fixing the bot: it's putting the whole order pipeline on infrastructure Hi-Service can rely on, and adding a web ordering channel + AI customer-service layer at the same time.

The bot today

122 n8n nodes on a third-party server, 38 hand-written Code nodes, hand-rolled MD5 in JavaScript, daily session resets at midnight, Google Sheets used as a slot-booking database (race conditions).

WhatsApp-only. No web shop. No admin panel. No conversation log. Customers without WhatsApp can't order.

The system we're building

Single PHP/MySQL backend on cPanel — same proven stack as Westpeak, Praxis and N24x4. Both WhatsApp and the new orders.hi-servicegas.co.za web shop run off it.

A GHL AI bot handles general help mode, calendar bookings, and admin escalation. The PHP backend handles every order.

The 2026 thesis

Stop fighting n8n. Stop fearing midnight resets. Open a web ordering channel. Give the team an admin panel that shows live orders + live chats.

Use GHL AI as a 24/7 customer-service rep so Lele's team only handles real escalations.

Critical findings from the n8n audit

1 · PayFast still pointed at sandbox

Live URL hardcoded as sandbox.payfast.co.za. If this branch fires, no real money moves. Sandbox merchant ID, key and passphrase are also embedded in two Code nodes, replicated in the PayFast sub-workflow.

Severity: Blocker for any cutover.

2 · Duplicate state definition kills the new-customer branch

The state-machine controller defines confirm_new_details twice. The second definition overwrites the first. Brand-new customers can land in a dead branch — almost certainly the source of the production "issues".

3 · Daily session reset at midnight

Session key is phone:yyyy-MM-dd. A customer who starts an order at 23:55 loses all state at 00:00 — order details, address, slot selection, gone. Silent UX failure.

4 · Google Sheets as a scheduling database

Slot bookings read/write across 9 separate Sheets nodes. Two simultaneous bookings can both succeed against the same slot. No atomic locking. No audit trail.

5 · Pure regex intent detection

Inputs like "1.", "yes please", "i'd like to order" all fall through to "unclear". Zero AI fallback. Customers who deviate from the script get stuck.

6 · No global error handler · no conversation log

If any of the 38 Code nodes throws, execution dies and the customer never gets a reply. Past messages are not retained anywhere queryable — operators can't debug "what did this customer say at 3pm".

Stage 1 · backend skeleton

cPanel · DB schema · auth · admin shell

Days1 – 2

Subdomain + DB + auth

Provision orders.hi-servicegas.co.za on the hivecliqapps cPanel. AutoSSL. New MySQL DB hiservicegas_main. Schema for customers, products, slots, orders, order_lines, conversations, sessions, payments. Branded login page + session auth.

Demo: log in to a working empty admin panel.
Stage 2 · the brain

State machine · intent detection · PHPUnit tests

Days3 – 4

Port FSM from JS to PHP — fix the bugs

Translate the existing 13-state machine into a clean PHP class. Fix the duplicate confirm_new_details definition. Replace daily session reset with a 24-hour rolling expiry. Optional Claude AI fallback when regex misses an intent. Full PHPUnit test coverage on every state transition.

Demo: run unit tests · simulate any conversation in a CLI.
Stage 3 · external integrations

Meta Graph · Xero · PayFast · GHL

Days5 – 8

Day 5 — Meta WhatsApp Cloud API gateway

Webhook receiver with Meta signature verification. Send helper using Graph API. Round-trip tested with a real test number.

Day 6 — Xero contacts + invoices

OAuth refresh token flow. Contact create/update/find. Invoice create with line items. Invoice PDF download.

Day 7 — PayFast pay link + ITN

PHP md5() signature (one line, replaces 100 lines of JS). Pay-link generator. ITN webhook with signature verify and amount cross-check.

Switch: sandbox → live merchant credentials.

Day 8 — GHL bridge

Forward general-help messages into GHL Conversations API. Receive outbound webhook from GHL workflow (AI reply). Send back to WhatsApp.

Stage 4 · web shop + admin

orders.hi-servicegas.co.za · live ops dashboard

Days9 – 12

Days 9–10 — public web ordering shop

Six-step funnel: postal-code check · identify (phone/email) · browse products · confirm address · pick slot · pay. Same MySQL slots table as WhatsApp — no double bookings. Mobile-first responsive design on Hi-Service brand.

Days 11–12 — admin panel

Live orders board with status. Live conversations table with "take over" button per chat. Slot calendar editor. Product catalogue editor (replaces the Google Sheet whitelist). Reports: revenue, slot fill rate, conversation outcomes.

Stage 5 · cutover

End-to-end test · go live · old n8n stays standby

Days13 – 14

Day 13 — full dress rehearsal

Real WhatsApp number → real PayFast → real Xero → real PDF delivered. Verify both lanes (gas + general help) end-to-end. Run a load test on slot booking concurrency.

Day 14 — flip the webhook

Switch the Meta WhatsApp webhook URL from n8n.imappliedseo.com to the new orders.hi-servicegas.co.za/api/webhook/whatsapp.php. Same for PayFast ITN. Old n8n bot stays running on standby for 7 days as rollback. Decommission once stable.

Outcome: live cutover with same-day rollback option.

A hybrid. Not a choice.

PHP for the parts where deterministic order processing matters. GHL for the parts where conversational AI and CRM matter. Same proven pattern as Westpeak (PHP + GHL) and HiveCliq Command Center (Next.js + Supabase + GHL).

PHP backend owns

  • WhatsApp webhook + state machine
  • Web ordering shop (orders.hi-servicegas)
  • MySQL: orders · slots · conversations
  • Xero invoice creation
  • PayFast pay link + ITN verification
  • Admin panel + reports
  • Conversation logs (every message)
  • Take-over button (human override)

GHL owns

  • Customer-service AI bot (FAQ)
  • Calendar booking flows
  • Admin escalation notifications
  • CRM contact record (synced)
  • Email + SMS sequences
  • Newsletter capture
  • Workflow triggers
  • Reporting on AI conversation outcomes

How they bridge

  • Backend forwards inbound text to GHL via API
  • GHL workflow runs AI bot · books · escalates
  • GHL fires outbound webhook back to backend
  • Backend sends the reply via Meta Graph
  • Every order syncs to GHL contact record
  • Escape word "menu" works from any state
  • Single Meta WhatsApp number for both lanes
